# Contributing to Startup OS

Thank you for your interest in contributing to Startup OS! This project aims to provide high-quality, production-ready templates that help founders build their companies faster and more efficiently.

## Vision

Startup OS is a comprehensive collection of templates, documents, and frameworks that cover every aspect of building and running a startup. Our goal is to:

- Save founders time by providing battle-tested templates they can use immediately
- Establish best practices across all startup functions (foundation, brand, product, finance, GTM, legal, hiring, operations, sales, metrics)
- Create consistency and professionalism in startup documentation
- Enable founders to focus on building their product rather than reinventing standard processes

## What We're Looking For

We welcome contributions that:

### High-Quality Templates
- Documents that solve real startup problems
- Templates that have been used successfully in actual companies
- Frameworks that scale from pre-seed through Series A and beyond
- Examples that demonstrate best practices in their domain

### Categories We Cover
- **01-foundation**: Company formation, vision/mission, values, culture docs
- **02-brand**: Brand guidelines, messaging, visual identity, voice/tone
- **03-product**: PRDs, roadmaps, feature specs, user research templates
- **04-finance**: Financial models, budgets, investor updates, metrics dashboards
- **05-go-to-market**: GTM strategy, launch plans, positioning, messaging
- **06-legal**: Contracts, agreements, compliance checklists (non-legal advice)
- **07-hiring**: Job descriptions, interview guides, onboarding plans
- **08-operations**: Process documentation, policies, operational frameworks
- **09-sales**: Sales playbooks, pitch decks, proposal templates
- **10-metrics**: KPI dashboards, reporting templates, analytics frameworks

### Improvements to Existing Templates
- Bug fixes or corrections
- Enhanced clarity or usability
- Additional examples or use cases
- Better formatting or structure

## Quality Standards

All contributions must meet our quality bar. Before submitting, ensure your work meets these criteria:

### Content Quality Checklist

- [ ] **Practical and Actionable**: Template can be used immediately with minimal modification
- [ ] **Complete**: All sections are filled out with meaningful content, no placeholders or TODO items
- [ ] **Professional**: Writing is clear, concise, and free of errors
- [ ] **Realistic**: Examples reflect real-world scenarios and constraints
- [ ] **Consistent**: Uses Helio as the example company (see style guide below)
- [ ] **Well-Structured**: Logical flow, clear headings, easy to navigate
- [ ] **Contextual**: Includes brief explanatory notes where helpful
- [ ] **Scalable**: Works for companies at different stages (note any stage-specific guidance)
- [ ] **Non-Promotional**: No product placements, affiliate links, or advertisements
- [ ] **Properly Licensed**: You have the right to contribute this content

### Technical Quality Checklist

- [ ] **Format**: Uses Markdown (.md) format with proper syntax
- [ ] **Naming**: Clear, descriptive filename using kebab-case (e.g., `product-requirements-document.md`)
- [ ] **Organization**: Placed in the correct category folder
- [ ] **Links**: All internal links work correctly
- [ ] **Rendering**: Previews correctly on GitHub
- [ ] **Metadata**: Includes a brief description at the top explaining what the template is for

## Style Guide

### Using Helio as the Example Company

To maintain consistency across all templates, we use **Helio** as our standard example company. This helps users quickly understand how templates relate to each other and see how different documents connect across the startup lifecycle.

#### Helio's Profile
- **Company**: Helio
- **Product**: AI-powered solar panel optimization platform
- **Market**: Commercial and residential solar installations
- **Stage**: Series A (raised $8M)
- **Team Size**: 25 employees
- **Founded**: 2023
- **Headquarters**: Austin, Texas

#### Example Roles at Helio
- **CEO**: Jordan Chen
- **CTO**: Alex Rivera
- **Head of Product**: Sam Park
- **Head of Sales**: Morgan Taylor
- **Head of Marketing**: Casey Williams

Feel free to elaborate on Helio's details as needed for your template, but maintain consistency with these core facts.

### Writing Style

- **Clear and Direct**: Use plain language, avoid jargon unless necessary
- **Action-Oriented**: Focus on what to do, not just what to think about
- **Specific**: Include concrete examples, numbers, and timeframes
- **Empathetic**: Remember that founders are busy and often doing this for the first time
- **Honest**: Acknowledge tradeoffs and challenges where they exist

### Formatting Conventions

- Use `#` for main title, `##` for major sections, `###` for subsections
- Include a brief description/overview at the top of each template
- Use bullet points for lists, numbered lists for sequential steps
- Use tables for structured data comparisons
- Use code blocks (```) for any code or formatted text
- Include a "How to Use This Template" section when helpful
- Add a "Last Updated" date at the bottom

## How to Contribute

### 1. Fork and Clone

Fork the repository and clone it locally:

```bash
git clone https://github.com/YOUR-USERNAME/startup-os.git
cd startup-os
```

### 2. Create a Branch

Create a branch with a descriptive name:

```bash
git checkout -b add-sales-proposal-template
```

### 3. Add Your Contribution

- Place your template in the appropriate category folder (01-foundation through 10-metrics)
- Follow the quality standards and style guide above
- Test that your Markdown renders correctly
- Ensure all links work

### 4. Commit Your Changes

Write clear, descriptive commit messages:

```bash
git add 09-sales/sales-proposal-template.md
git commit -m "Add sales proposal template with Helio example"
```

### 5. Submit a Pull Request

Push your branch and open a pull request:

```bash
git push origin add-sales-proposal-template
```

In your PR description:
- Explain what you're adding or changing
- Describe why it's valuable
- Note any decisions you made or questions you have
- Confirm you've checked the quality checklist

### PR Title Format

Use clear, descriptive titles:
- `Add [template name] for [category]`
- `Update [template name] to improve [aspect]`
- `Fix [issue] in [template name]`

Examples:
- `Add investor update template for finance`
- `Update PRD template to improve use case examples`
- `Fix formatting issues in brand guidelines template`

## Review Process

### What to Expect

1. **Initial Review**: A maintainer will review your PR within 5-7 days
2. **Feedback**: We may suggest changes to improve quality, clarity, or consistency
3. **Iteration**: Work with us to refine the contribution
4. **Approval**: Once the contribution meets our standards, we'll merge it
5. **Credit**: You'll be credited as a contributor

### Review Criteria

We review contributions based on:
- **Quality**: Does it meet our quality standards checklist?
- **Usefulness**: Will this help founders in a meaningful way?
- **Fit**: Does it align with our vision and existing content?
- **Consistency**: Does it follow our style guide (especially Helio examples)?
- **Completeness**: Is it ready to use, or does it need more work?

### Feedback Philosophy

We aim to be:
- **Constructive**: Feedback focuses on making the contribution better
- **Specific**: We'll point to exact issues and suggest solutions
- **Respectful**: We appreciate your time and effort
- **Collaborative**: We're here to help you succeed

## Questions?
